# وثيقة التصميم التقني: محرك البحث ونظام المرجعية والملاحظات

بناءً على متطلبات التطبيق المعمارية (Flutter, Riverpod, Clean Architecture)، تم إعداد هذه الوثيقة لتفصيل أفضل الاستراتيجيات التقنية لتنفيذ محرك البحث المتقدم ونظام المستخدم (المفضلة والملاحظات).

---

## 1️⃣ محرك البحث المتقدم (Advanced Search)

### أ. تحليل قواعد البيانات للبحث النصي
في Flutter، خيارات البحث النصي تنحصر بشكل رئيسي بين **SQLite (Drift/sqflite)** و **Isar** و **ObjectBox**.
- **Isar / ObjectBox:** سريعة جداً (NoSQL)، وتدعم البحث النصي، لكن قدراتها في الـ (Full-Text Search) المعقد والمطابقة الجزئية للنصوص العربية قد تكون محدودة مقارنة بـ SQLite.
- **SQLite (مع إضافة FTS5):** هي المعيار الذهبي للبحث النصي (Full-Text Search). محرك FTS5 مخصص خصيصاً للبحث السريع في النصوص الكبيرة، ويدعم المطابقة الدقيقة (MATCH)، وتجاهل اللواحق.

**🏆 الحل المرشح:** استخدام **SQLite عبر حزمة `drift`** مع تفعيل محرك **FTS5**. `Drift` توفر أمان الأنواع (Type-Safety) وتكامل ممتاز مع مجاري البيانات (Streams).

### ب. معالجة النصوص العربية (Text Normalization)
السر الحقيقي للبحث الدقيق في القرآن ليس في قاعدة البيانات فقط، بل في "تسوية النص" قبل حفظه وقبل البحث عنه.
يجب إنشاء جدول مخصص للبحث يحتوي على النص "المجرد" بجانب النص العثماني.

**خطوات الـ Normalization (خوارزمية تنظيف النص):**
1. إزالة جميع التشكيلات (الفتحة، الضمة، الكسرة، الشدة، التنوين... الخ).
2. توحيد أشكال الألف (أ، إ، آ، ٱ) لتصبح ألف مجردة (ا).
3. توحيد الياء والألف المقصورة (ى، ي) لتصبح (ي).
4. توحيد التاء المربوطة والهاء (ة، هـ) لتصبح (ه).

**آلية البحث:**
عندما يبحث المستخدم عن "الصلوه"، يتم تمرير الكلمة لنفس دالة الـ Normalization لتصبح "الصلوه" ويبحث عنها في عمود `search_text`.

## 2️⃣ نظام العلامات المرجعية، الوسوم، والملاحظات (Bookmarks, Tags & Notes)

### أ. اختيار قاعدة البيانات
بما أن هذا النظام يعتمد على العلاقات (Relations) (آية لها عدة وسوم، ووسم له عدة آيات "Many-to-Many")، فإن **`Drift` (SQLite)** هي الأنسب تماماً لربط الجداول، مما يجعلنا نستخدم `Drift` كقاعدة بيانات موحدة للبحث ولملفات المستخدم.

### ب. هيكل البيانات المقترح (Data Models / Schema)
سنقوم بإنشاء الجداول التالية باستخدام `Drift`:

```dart
// جدول الملاحظات
class Notes extends Table {
  IntColumn get id => integer().autoIncrement()();
  IntColumn get surahId => integer()();
  IntColumn get ayahId => integer()();
  TextColumn get text => text()();
  DateTimeColumn get createdAt => dateTime().withDefault(currentDateAndTime)();
}

// جدول العلامات المرجعية (المفضلة)
class Bookmarks extends Table {
  IntColumn get id => integer().autoIncrement()();
  IntColumn get surahId => integer()();
  IntColumn get ayahId => integer()();
  DateTimeColumn get createdAt => dateTime().withDefault(currentDateAndTime)();
}

// جدول الوسوم (Tags)
class Tags extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get name => text()();
  IntColumn get colorCode => integer()(); // لحفظ لون الوسم
}

// جدول الربط بين الآيات والوسوم (Many-to-Many)
class AyahTags extends Table {
  IntColumn get tagId => integer().references(Tags, #id)();
  IntColumn get surahId => integer()();
  IntColumn get ayahId => integer()();
  
  @override
  Set<Column> get primaryKey => {tagId, surahId, ayahId};
}
```

### ج. التكامل مع Riverpod للتحديث الفوري (Reactive UI)
لتجنب إعادة بناء (Rebuild) صفحة المصحف بالكامل عند وضع علامة مرجعية على آية، سنستفيد من ميزة Streams في Drift مع `Provider.family` في Riverpod.

1. **في طبقة Data:** الدالة في `Drift` تُرجع `Stream<bool>` يراقب ما إذا كانت الآية في المفضلة.
2. **في طبقة Application (Riverpod):**
```dart
// Provider مخصص لمراقبة حالة آية محددة فقط
final isAyahBookmarkedProvider = StreamProvider.family<bool, String>((ref, ayahKey) {
  final repository = ref.watch(userDataRepositoryProvider);
  return repository.watchIsBookmarked(ayahKey); // يعود بـ Stream من Drift
});
```
3. **في طبقة Presentation:** 
نغلف أيقونة العلامة المرجعية (Bookmark Icon) الخاصة بكل آية بـ `Consumer` يقرأ `isAyahBookmarkedProvider(ayahKey)`.
عندما يقوم المستخدم بحفظ الآية، تتحدث الـ Database، فتُصدر Drift حدثاً جديداً عبر الـ Stream، فيقوم Riverpod بتحديث **أيقونة هذه الآية فقط** (O(1) Rebuild) دون المساس بباقي صفحة المصحف.

## 3️⃣ توزيع الميزات داخل "البنية النظيفة" (Clean Architecture)

للحفاظ على قابلية التوسع والاختبار، يتم توزيع المكونات كالتالي:

1. **الطبقة الأساسية (Data Layer):**
   - `DriftDatabase`: إعداد الجداول (Tables) والـ DAOs.
   - `QuranLocalDataSource`: ينفذ استعلامات FTS5 والبحث.
   - `UserDataLocalDataSource`: ينفذ الإضافة والحذف للوسوم والملاحظات.

2. **طبقة النطاق (Domain Layer):**
   - **Entities:** كائنات نقية مثل `NoteEntity`, `TagEntity`, `SearchResultEntity`.
   - **Repositories Interfaces:** `ISearchRepository`, `IUserDataRepository`.
   - **Use Cases:** كلاسات تقوم بمهمة واحدة، مثل: `SearchQuranUseCase`, `AddNoteUseCase`, `GetTagsForAyahUseCase`. هنا نضع منطق الـ Text Normalization قبل إرسال البحث لطبقة الداتا.

3. **طبقة التطبيق (Application / State Layer):**
   - **Riverpod Providers:** `searchControllerProvider` (StateNotifier أو AsyncNotifier لإدارة حالة شريط البحث ونتائجه)، و `tagsNotifierProvider`.

4. **طبقة العرض (Presentation Layer):**
   - `SearchScreen`: واجهة البحث مع شريط يراقب الـ SearchController.
   - `AyahInteractionBottomSheet`: القائمة السفلية التي تظهر عند الضغط المطول على الآية وتحتوي أزرار (إضافة ملاحظة، حفظ مرجعي).
